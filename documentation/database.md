# Database Architecture

## Overview
The application relies on PostgreSQL provided by **Supabase**. The database acts as the single source of truth and enforces authorization via Row Level Security (RLS) policies.

## Key Schemas and Configurations
- **Extensions**: `uuid-ossp` is utilized across all tables to enforce globally unique identifiers.
- **Enums**: Custom PostgreSQL enums are used to enforce type safety at the database level for fields like `role_type` ('COACH', 'ATHLETE'), `training_type`, `metric_type`, and `sync_status`.

## Core Entities

### 1. Identity and Profiles
- `profiles`: The central table mapped 1:1 with `auth.users`. It stores the `role` ('COACH' or 'ATHLETE') and a `coach_id` defining the mentor relationship.
- `coach_profiles`: Additional metadata for coach users (bio, experience, specialties).
- `athlete_profiles`: Health metrics and demographic data pertinent to athletes (weight, max HR, VAM/UAN metrics).
- `athlete_metrics`: A historical log of metric updates (like new VAM tests) over time.

### 2. Trainings and Assignments
- `trainings`: The library of workout templates. Includes a `blocks` JSONB column holding the complex interval structures.
- `training_assignments`: The binding of a `training` to a specific `user_id` (athlete) on a `scheduled_date`. Contains the compliance status and coach feedback.

### 3. Groups
- `groups`: Logical collections of athletes managed by a Coach. Can include target race dates.
- `athlete_groups`: The junction table resolving the many-to-many relationship between `profiles` (athletes) and `groups`.

### 4. Third-Party Data (Strava)
- `activities`: Stores the synchronized workouts imported from Strava (distance, time, elevation, pace).
- `strava_activity_streams`: Granular, second-by-second data streams (Heart Rate, Coordinates, Pace) stored as heavy JSONB payloads.
- `strava_connections`: Highly sensitive table housing OAuth tokens (`access_token`, `refresh_token`) required to poll the Strava API on behalf of the user.

### 5. Utility Tables
- `invitations`: Secure, time-limited tokens generated by coaches to onboard new athletes via email.
- `sync_logs`: Tracks the health and status of background processes syncing data with Strava.

## Security Model: Row Level Security (RLS)
The database restricts access based on policies. Some fundamental RLS rules applied:
- **Athletes** can only SELECT/UPDATE their own rows in `profiles`, `activities`, and `training_assignments`.
- **Coaches** can query `training_assignments` and `activities` for any user that is linked to them via the `athlete_groups` table.
- **Strict Isolation**: `strava_connections` is locked down such that tokens are only readable by the system or the exact authenticated owner to prevent token leakage.
